# ResearchPilot AI - Internal Project Specification v1.0

This document serves as the single source of truth for the architecture, implementation details, schemas, and design guidelines of the **ResearchPilot AI** application.

---

## 1. Project Overview

### Project Name & Tagline
- **Project Name:** ResearchPilot
- **Tagline:** Research Smarter. Verify Faster. Build Better.

### Project Vision
To build the **GitHub + Notion + Perplexity** for academic research. ResearchPilot aims to become the primary workspace environment where students, researchers, and professors can ingest, analyze, critically evaluate, plan, and discuss scientific papers in one place.

### Problem Statement
Traditional academic research workflows are fragmented and suffer from:
- **Information Overload:** Researchers struggle to filter and process thousands of long papers.
- **Manual Review Constraints:** Synthesizing methodologies and extraction plans takes weeks.
- **Weak AI Grounding:** standard ChatGPT integrations hallucinate metrics and lack verifiable citation bounds.
- **Disconnected Workspaces:** Reading PDFs, drafting implementation roadmaps, and chatting with documents happen in separate, unsynchronized tools.

---

## 2. Product Philosophy

### Trust Through Citations
Every claim, suggestion, or summary generated by the AI must be auditable. In-text citation chips map directly to a persistent, interactive references sidebar. Users can click any citation to focus instantly on the verified source fragment.

### Academic First Design
The visual aesthetic is premium, minimal, calm, and academic. It prioritizes reading legibility (using a Serif font for long-form content, Sans-Serif for controls, and Monospace for statistics) over loud gradients, heavy shadows, or consumer SaaS clutter.

### Low Cognitive Load
Focuses the interface on a single major workflow at a time. Secondary details remain collapsed by default using **progressive disclosure** (accordions and toggles).

### Research Workflow Focus
Aligns user flows to the natural scientific inquiry path:
$$\text{Ingest/Upload} \longrightarrow \text{Analyze Summary} \longrightarrow \text{Evaluate Review} \longrightarrow \text{Replicate Action Plan} \longrightarrow \text{Inquire via Chat}$$

## 4. Feature Breakdown

### Dashboard
- **Purpose:** Serve as a Research Control Center showcasing library summaries and active workspaces.
- **User Flow:** Open landing dashboard -> review recent papers table -> click resume research -> jump to the active paper workspace.
- **Frontend Flow:** TanStack Query `usePapersList` and `useDashboardStats` fetch database rows.
- **Backend Flow:** Exposes paper metadata and aggregate statistics from SQLite.

### Ingestion (Upload)
- **Purpose:** Process documents, generate index vectors, and catalog metadata.
- **User Flow:** Ingest paper -> review text extraction, chunking, and vector database populate statuses -> receive success metadata -> open workspace.
- **Backend Flow:** Validates files, saves to `uploads/`, processes pages via PyMuPDF, chunks text, generates embeddings, uploads vectors, and writes parameters to SQLite.

### Analyze
- **Purpose:** Fast document summarization using progressive disclosure.
- **User Flow:** Select Analyze tab -> review authors and TLDR -> expand accordion sections (Problem Statement, Methodology, Results, Gaps).

### Review
- **Purpose:** Critically assess the paper.
- **User Flow:** Select Review tab -> check Quality score badges (reproducibility checks, validity indexes) -> review strengths and weaknesses.

### Action Plan
- **Purpose:** Replicate and implement research.
- **User Flow:** Select Action Plan tab -> review steps -> check off completed milestones -> add logs and parameter configurations in textareas.

### Chat
- **Purpose:** Inquire and converse with the document.
- **User Flow:** Select Chat tab -> input query or select suggested prompt -> AI retrieves matching context -> click citation chips to highlight source text.

## 7. AI & Vector Search Architecture

### Text Ingestion & Chunking
When a PDF is uploaded, `pdf_processor.py` extracts raw text page-by-page. The text is passed to `text_chunker.py` which:
- Splits text into segments up to **1000 characters**.
- Implements a **200-character overlap** to preserve semantic continuity across borders.
- Searches backwards from boundaries to split at sentence endings (`.`, `!`, `?`), paragraph breaks (`\n\n`), or space limits.

### Embedding Generation
The text segments are embedded into vectors using `SentenceTransformer` loading the `all-MiniLM-L6-v2` model (producing **384-dimensional** dense vectors).

### Vector Storage (ChromaDB)
Vectors are uploaded to a local ChromaDB persistent store collection (`research_papers` by default). Each chunk is stored with metadata:
```json
{
  "file_id": "uuid-string",
  "chunk_index": 0
}
```

### Retrieval Augmented Generation (RAG)
When a query is submitted in Chat:
1. The search string is embedded via the embedding service.
2. ChromaDB queries for the **top 5 closest chunks** filtered by `"where": {"file_id": file_id}`.
3. The retrieved chunks are structured into context strings: `[Source 1]: text_content`.
4. A RAG prompt is assembled and sent to Gemini:
   ```
   Context from the research paper:
   [Source 1] ...

   Question: [User Question]

   Answer the question using only the provided context...
   ```
5. Returns the response with the average cosine similarity metric as a confidence score.

## 8. Database Architecture

### SQLite Metadata Store
Manages metadata cataloging, statuses, and statistics.
- **Table Name:** `papers`
- **Fields:**
  - `id` (TEXT, Primary Key): UUID file identifier
  - `title` (TEXT): Title of the research paper
  - `authors` (TEXT): Author names
  - `year` (TEXT): Publication year
  - `upload_time` (TIMESTAMP): Ingestion timestamp
  - `file_path` (TEXT): Location of the PDF file on disk
  - `research_health` (TEXT): Health state (`"Fully Analyzed"`, `"Summary Only"`, `"Not Reviewed"`)
  - `favorite` (BOOLEAN): Defaults to `0`
  - `page_count` (INTEGER): Total pages in the PDF
  - `reading_time_minutes` (INTEGER): Calculated reading duration

## 9. API Documentation

### Ingestion & Uploads
- **`POST /api/v1/upload/pdf`**
  - **Request:** Multipart Form-data (`file: UploadFile`)
  - **Response:**
    ```json
    {
      "file_id": "string",
      "filename": "string",
      "file_size": 1024,
      "status": "uploaded",
      "uploaded_at": "datetime-string",
      "title": "string",
      "authors": "string",
      "page_count": 12,
      "reading_time_minutes": 15
    }
    ```

### Library Catalog & Stats
- **`GET /api/v1/papers`**
  - **Response:** `{ "success": true, "data": Array<Paper> }`
- **`GET /api/v1/papers/stats`**
  - **Response:** `{ "success": true, "data": { "total_papers": int, "analyses_generated": int, "active_workspaces": int, "favorites": int } }`

### Document Analysis
- **`GET /api/v1/analysis/{file_id}/summary`**
  - **Response:**
    ```json
    {
      "success": true,
      "data": {
        "file_id": "string",
        "summary": {
          "problem": "string",
          "method": "string",
          "results": "string",
          "limitations": "string",
          "future_work": "string"
        }
      }
    }
    ```

- **`GET /api/v1/analysis/{file_id}/review`**
  - **Response:** Returns critical assessments, reproducibility checkmarks, strengths, and weaknesses.

- **`GET /api/v1/analysis/{file_id}/action-plan`**
  - **Response:** Returns skills, learning paths, project extensions, and resources.

- **`POST /api/v1/chat/{file_id}`**
  - **Request:** `{ "question": "string" }`
  - **Response:** `{ "success": true, "data": { "answer": "string", "confidence": float, "sources": Array<Source> } }`

## 10. Complete User Journey

1. **Upload Phase:** User drags a PDF onto the Upload Page. The app extracts text, generates embeddings, catalogs parameters, and runs SQLite migrations.
2. **Reviewing the Summary:** The user opens the Workspace, landing on the **Analyze** tab. The summary outlines the Problem Statement, Methodology, and Results.
3. **Evaluating Methodology:** The user switches to the **Review** tab to see Quality checks (reproducibility scores) and critical strengths/weaknesses.
4. **Implementation Planning:** The user views the **Action Plan** tab to see step-by-step implementation roadmaps and resources.
5. **Interactive Questioning:** The user uses the **Chat** tab to ask specific questions. The RAG engine answers using the vector database, and the user can click citation chips to highlight sources in the references panel.

## 12. Setup and Installation

### Prerequisites
- Python 3.10+
- Node.js 18+

### Backend Setup
1. Navigate to the backend directory:
   ```bash
   cd backend
   ```
2. Create and activate a virtual environment:
   ```bash
   python -m venv venv
   source venv/bin/activate  # On Windows: venv\Scripts\activate
   ```
3. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```
4. Start the FastAPI server:
   ```bash
   uvicorn app.main:app --reload
   ```

### Frontend Setup
1. Navigate to the frontend directory:
   ```bash
   cd frontend
   ```
2. Install dependencies:
   ```bash
   npm install
   ```
3. Start the Vite development server:
   ```bash
   npm run dev
   ```
